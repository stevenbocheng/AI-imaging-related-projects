# 🚀 Hyper-Math Vision 專案現況總覽

這是一個基於電腦視覺與生成式 AI (Gemini) 的「大空間手勢運算平台」。使用者可以直接對著 Webcam，在虛擬空間中舉起手指書寫數學算式，並透過手勢控制將畫面送交給 AI 獲得解答。

## 🎯 核心互動功能 (Features)
1. **凌空書寫 (Air Writing)**
   - 伸出**單根食指**：觸發畫筆功能，將軌跡即時繪製於虛擬畫布上。
2. **手勢擦拭 (Air Eraser)**
   - 伸出**單根大拇指**：啟動清除模式（需穩定維持約 0.5 秒），一鍵清空整個虛擬黑板。
3. **智慧送出解答 (Smart Solve)**
   - 伸出**四根手指/五指全開**：觸發截圖並自動裁切、優化，將算式傳輸至 Gemini API（內建 3 秒防誤觸冷卻），隨後在 UI 介面上即時反饋數學運算結果。
4. **背景分離繪圖**
   - 採用雙層畫布設計：底層顯示使用者的真實相機影像（透明度 0.7 疊加），強調擴增實境感；頂層為乾淨的黑底紫線畫布，專門用來擷取純淨的高對比數學公式。

# 🧠 實作的硬核演算法 (CV & Signal Processing Algorithms)
在專案的升級重構中，為了解決一般手勢操作中最常見的「抖動、斷線、環境光線干擾、透視變形」等痛點，我們整合了以下核心演算法與機制：

### 1. 3D 深度投影補償 (3D Perspective Compensation)
* **技術痛點**：原本使用 2D 像素座標 (`lmList`) 以及單純的「指尖高度是否大於指根高度」來判斷開合。當手掌方向未與鏡頭平行時（例如俯視或仰視），因為透視錯覺，原本伸直的手指在 2D 畫面上會變短，導致嚴重誤判。
* **演算法解法**：改擷取 MediaPipe 提供的 `World Landmarks`。這套座標系統會將相機畫面逆向映射回 **以公尺為單位的真實 3D 空間網格**。我們利用 3D 向量運算（內積與反餘弦 `np.arccos(cosine)`）算出真實的骨骼關節空間夾角，讓手勢判定完全「免疫」相機的透視變形。

### 2. 空間歸一化尺規 (Spatial Normalization via Palm Reference)
* **技術痛點**：當使用者手跟鏡頭距離極近時，相機不可避免會產生邊緣魚眼畸變，導致部分 3D 角度的計算依然會失效。
* **演算法解法**：導入自適應尺規概念。以當下抓取到的手掌心長度（手腕到中指根座的 3D 歐式距離）作為基準長度 `palm_ref`。再將食指伸出的絕對長度除以 `palm_ref`。如此一來，這個 `finger_length` 特徵值就具備了 **「尺度不變性 (Scale Invariance)」**，解決了太靠近鏡頭無法辨識的問題。

### 3. 時間窗狀態機與防呆計數 (Time-window State Machine)
* **技術痛點**：原本程式將邏輯視為「無狀態 (Stateless)」的直接反應，導致使用者在書寫過程中，若切換手勢瞬間不小心閃過「大姆指」，就會在一幀（0.03秒）內把畫布意外清空。
* **演算法解法**：建立微型狀態機。系統會暫存上一幀的手勢陣列，若本幀手勢與上一幀相同，才允許狀態計數器累積。在執行破壞性指令（如清除、送出 API）前，強制要求該狀態必須連續維持超過 15 幀或 20 幀。將時間維度加入判定，徹底消除動作轉換過程中的誤觸。

### 4. 遲滯雙閾值過濾 (Hysteresis Filtering / Deadband)
* **技術痛點**：如果只設定單一及格線（例如 >150 度代表伸直，<150度代表彎曲）。當使用者手指保持在 150 度微弱抖動時，程式會以一秒 30 次的頻率在斷斷續續畫線，也就是常見的「電子雜訊震盪」。
* **演算法解法**：引入工業控制中的「遲滯比較器」。我們把「張開」的門檻拉高到 160 度（要求刻意伸直）；但把「收起」的門檻放寬到 135 度。兩者之中的區域（135° ~ 160°）視為緩衝死區 (Deadband)。如果使用者已經在書寫狀態，就算手痠抖回到 140 度，系統依然判定為書寫中。

### 5. 卡爾曼濾波器軌跡平滑 (Kalman Filter Trajectory Smoothing)
* **技術痛點**：相機擷取的指尖網格都是離散的整數像素。直接把上一幀跟這一幀連起來，畫面上會出現大量鋸齒。如果當下手抖或 MediaPipe 模組抽風算出極端座標，筆跡就會瞬間折斷或「牽絲」。
* **演算法解法**：建立具備物理慣性模型的卡爾曼濾波器。設定二維等速運動的轉移矩陣 (Transition Matrix)。每收到一個新座標，我們並不盲目信任相機，而是讓濾波器的「物理預測步 (Predict)」與相機的「觀測步 (Correct)」交給系統進行最優化融合，最終渲染出無抖動、高滑順度的字體軌跡。

### 6. 動態 ROI 裁剪引擎 (Dynamic Region-of-Interest Cropping)
* **技術痛點**：如果把高解析度的原始 1280x800 黑圖丟給生成式 AI，由於有效字體所佔面積比例極小，AI 非常容易辨識錯誤或是耗費大量運算資源。
* **演算法解法**：截圖送出前，程式會呼叫 OpenCV 自動進行灰階轉換與連通區域運算，在一瞬間掃描矩陣內所有亮點（白像素），計算出完美的邊界框 (Bounding Box)。再加上一點外擴邊緣，程式會自動用「剪刀」把核心算式部位切下來丟給 API。大幅提升辨識成功率以及降低網路傳輸頻寬需求。
